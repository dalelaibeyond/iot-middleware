MQTT Topic Format

The middleware uses a standardized topic hierarchy for MQTT messages: 

<category>/<gatewayId>/<type>

Example topics:

  • sensors/gateway123/temperature - Temperature sensor data
  • sensors/gateway123/humidity - Humidity sensor data
  • devices/gateway456/status - Device status updates
  • devices/gateway456/control - Device control messages

Message Relay System

The middleware includes a message relay system that republishes normalized messages to new topics:

Relay topic format: 

<category>/<prefix>/<gatewayId>/<type>

For example:

  • Original: sensors/gateway123/temperature
  • Relayed: sensors/new/gateway123/temperature

Configuration in config.json: 

{
    "messageRelay": {
        "enabled": true,
        "topicPrefix": "new",
        "patterns": {
            "sensors": "sensors/${prefix}/${gatewayId}/${type}",
            "devices": "devices/${prefix}/${gatewayId}/${type}"
        }
    }
}

Benefits:

  • Separate normalized data stream
  • Easy integration with other systems
  • Configurable topic patterns
  • Can be enabled/disabled as needed

Core Framework

Built on a modern, event-driven architecture with:

  • sensors/gateway123/temperature - Temperature sensor data
  • sensors/gateway123/humidity - Humidity sensor data
  • devices/gateway456/status - Device status updates
  • devices/gateway456/control - Device control messages

Topic components:

  • category: Either "sensors" or "devices"
  • gatewayId: Your gateway's unique identifier
  • type: The sensor type or message type

Resilience Features

Designed for reliability under high load:

  • Automatic Circuit Breaking for error prevention
  • Intelligent Rate Limiting for traffic control
  • Smart Batching for optimal performance
  • Resource-aware scaling

Features

This middleware service bridges MQTT-enabled IoT devices with applications by:

  • Supporting multiple MQTT topic subscriptions
  • Normalizing raw MQTT messages into a unified JSON format
  • Storing the latest readings in memory for fast access
  • Persisting historical data in MySQL with write buffering
  • Providing REST APIs for data access
  • Supporting real-time WebSocket updates
  • Offering HTTP callbacks with retry mechanism

Plugin System

Extend functionality through plugins:

// Example plugin
module.exports = {
    initialize: async (eventBus) => {
        eventBus.subscribe('message.received', 
            async (message) => {
                // Custom processing
            });
    }
};

Message Format

    

        
Input Example (Raw MQTT Message)

        
{
  "devId": "A-Temp-001",
  "rackNo": "42",
  "posU": "18",
  "tmp": 27.3,
  "unit": "C",
  "time": 1695653130000
}


        
Normalized Format

        
{
  "deviceId": "string",        // unique device id
  "sensorType": "string",      // e.g. "temperature"
  "ts": "2025-09-24T08:12:23.456Z", // ISO8601 UTC timestamp
  "seq": 12345,                // optional sequence number
  "payload": {                 // sensor-specific data
     "temp": 28.3,
     "unit": "C",
     "rackNo": "42",
     "posU": "18"
  },
  "meta": {                    // metadata
     "rawTopic": "sensors/gateway-1/temp",
     "gatewayId": "gateway-1"
  }
}
